The newsroom is the brand's owned PR surface — the page on the
brand's domain where press releases archive, the media kit
downloads, executive bios live, and journalists find press
contact info.

This command produces the **content manifest** for the newsroom
page. The actual page design is generated by
`/product:design:templates:newsroom` (which reads this content manifest).
Two-step separation matches the rest of the suite: content
production and design production are distinct concerns.

## What the newsroom page contains

```
┌─────────────────────────────────────────────────────────┐
│ Newsroom hero                                           │
│   - "{{Brand}} Newsroom" or similar                    │
│   - Tagline + press contact prominent                  │
│   - Media-kit download CTA                             │
├─────────────────────────────────────────────────────────┤
│ Recent press releases (chronological, paginated)       │
│   - Latest 6-12 with summaries on this page           │
│   - Older releases linked to archive                  │
│   - Each release: date, headline, lead paragraph,     │
│     "Read more" link to full release page             │
├─────────────────────────────────────────────────────────┤
│ Media kit                                              │
│   - Logos, headshots, fact sheet, brand guidelines    │
│   - Download as zip OR browse individual files        │
├─────────────────────────────────────────────────────────┤
│ Executive bios                                         │
│   - Spokesperson registry rendered as bio cards       │
│   - Photo + title + short bio + topics they speak on  │
├─────────────────────────────────────────────────────────┤
│ Press contact                                          │
│   - Name, title, email, phone                         │
│   - Embedded "Get in touch" form                      │
├─────────────────────────────────────────────────────────┤
│ Footer (standard chrome)                              │
└─────────────────────────────────────────────────────────┘
```

## Pre-flight

1. Read `product/strategy/_context.md`, `market/_context.md`,
   `market/pr/_context.md`, and (when established)
   `product/.pencil-seo.json`.
2. Read `product/.pencil-brand.json`.
3. Read `design/marketing/pr/boilerplate.json` (required —
   spokesperson registry + press contact source).
4. Walk `design/marketing/pr/releases/` for all release files.
5. Read `design/marketing/pr/media-kit/manifest.json` if it
   exists (when the media kit has been built; the newsroom
   links to it).
6. Resolve mode:
   - `generate` (default): full content manifest production
   - `refresh`: incremental — only update if releases or
     boilerplate changed since last generation
7. Resolve flags:
   - `--release-limit <n>` — how many recent releases to
     surface on the main newsroom page. Default 8. Older
     releases archive separately (paginated).
   - `--include-media-kit` — flag, default true. Include the
     media-kit download section in the manifest.
   - `--out <path>` — output path for the content manifest.
     Default `design/marketing/pr/newsroom-content.json`.
   - `--dry-run` — preview without writing.

## Phase 2 — Walk releases and pull lead paragraphs

For each release file in `design/marketing/pr/releases/*.md`,
extract:

- Slug (from filename)
- Type (from companion .json metadata)
- Date (from companion .json metadata `distributionDate`)
- Headline (first H1 in the .md file)
- Subhead (first paragraph after H1, before the dateline)
- Lead paragraph (first paragraph after the dateline)
- Spokesperson name (first quoted spokesperson in the release)

Sort by date descending. Take the first `--release-limit`
releases for `recentReleases`. Older releases reference the
archive page.

## Phase 3 — Cross-link with media kit

When media-kit's `manifest.json` exists, populate the mediaKit
section with:

- Last-built date
- Total size
- Download URL (zip when bundled; archive URL when folder)

When media-kit hasn't been built, surface a recommendation:
"Run `/market:pr:media-kit` to build the downloadable
press kit before deploying the newsroom."

## Phase 4 — Generate RSS feed manifest

Press releases are RSS-amenable content. Generate
`design/marketing/pr/newsroom-rss.json` describing the RSS feed:

```jsonc
{
  "version": 1,
  "feedUrl": "/newsroom/rss.xml",
  "title": "{{Brand}} Newsroom",
  "description": "Press releases and announcements from {{Brand}}",
  "language": "en-US",
  "items": [
    {
      "guid": "saved-searches-feature-launch-q2-2026",
      "title": "Acme launches saved searches for engineering teams to cut hours from weekly workflow",
      "description": "{{lead paragraph}}",
      "link": "https://acme.com/newsroom/releases/saved-searches-feature-launch-q2-2026",
      "pubDate": "Thu, 15 May 2026 09:00:00 -0400",
      "category": "product-launch"
    }
    // ... all releases (RSS includes the full archive, not just recent)
  ]
}
```

The actual RSS XML emission happens at the deploy layer (when
the newsroom HTML is built); this manifest tells the build
pipeline what to include.

## Phase 5 — SEO + AIO contract for newsroom

The newsroom page is a high-leverage SEO surface:

- **Branded keyword domination** — "{{Brand}} newsroom",
  "{{Brand}} press", "{{Brand}} news" should rank #1 on Google
  for branded queries
- **AI search citation** — when users ask AI search "what has
  {{Brand}} announced recently?", the newsroom page is the
  authoritative source

When `.pencil-seo.json` exists, the newsroom page's archetype
target is `newsroom` (a new archetype distinct from `marketing`):

```jsonc
"perArchetypeTargets": {
  "newsroom": {
    "primaryKeyword": "{{Brand}} newsroom",
    "headingHierarchy": "h1 (newsroom title) + h2 sections (recent releases, media kit, executives, press contact) + h3 per individual release in list",
    "structuredData": ["Organization", "NewsArticle", "BreadcrumbList"],
    "metaDescriptionLength": "150-160",
    "wordCountTarget": "varies by release count",
    "aioPatterns": ["faq-schema (when FAQ added)", "date-stamped-facts", "citation-ready"],
    "internalLinksMin": 8
  }
}
```

When the strategy doesn't have a `newsroom` archetype configured,
the newsroom command surfaces a recommendation to add it (or
use `marketing` archetype as fallback).

## Phase 6 — Hand off to template

The newsroom content manifest is the input to
`/product:design:templates:newsroom` (a new template — see Session 3-A
note below). The template reads the manifest and produces the
designed newsroom page.

Output the final report with explicit handoff:

```
✓ Newsroom content manifest generated.

Files:
  design/marketing/pr/newsroom-content.json    (content manifest)
  design/marketing/pr/newsroom-rss.json        (RSS feed manifest)

Recent releases:    8 (from 12 total in releases folder)
Media kit:          linked (last built 2026-05-02)
Executives:         2 from boilerplate registry
Press contact:      Sam Smith (press@acme.com)

Action items:
  1. Generate the newsroom page design:
     /product:design:templates:newsroom
     (reads newsroom-content.json; produces design/templates/newsroom.pen)
  2. Build the React/HTML for the newsroom:
     /frameworks:heroui:build-components design/templates/newsroom.pen
  3. Verify SEO contract for newsroom archetype:
     /audit --plane 9 (will catch missing archetype targets)
  4. Set up RSS feed at /newsroom/rss.xml from the rss manifest
     (deploy-layer concern; out of suite scope)
  5. Update boilerplate.json's currentStats if any are stale
```

## Session 3-A note — newsroom template

The `product/design/templates/newsroom.md` file (the design template that
consumes this command's output) is part of Session 3-A's scope
per user confirmation in the planning conversation. Until that
template exists, this command's output can be consumed by:

- `product/design/templates/marketing.md` with a custom variant
- Hand-coded newsroom page that reads the manifest

The newsroom template (when added) will:
- Render the three-pane layout (or single-column variant) per
  the manifest's structure
- Apply the newsroom archetype's SEO + AIO contract
- Compose with existing patterns (hero, CTA, footer)
- Produce three breakpoint frames per the suite's standard

## Idempotency

`generate` overwrites the manifest. `refresh` checks mtimes —
only regenerates when releases or boilerplate have changed.

The deploy-layer pipeline should regenerate the newsroom HTML
on every release-deploy, so adding a new release auto-updates
the newsroom without manual command runs. The content manifest
is the trigger.

## What this command does NOT do

- **Does not generate the newsroom page design.** The .pen
  design comes from `/product:design:templates:newsroom`.
- **Does not deploy the newsroom.** Static-site or app-deploy
  pipeline handles HTML emission and CDN.
- **Does not generate the RSS XML.** The XML emission is build-
  layer; this command produces the RSS content manifest.
- **Does not handle archive pagination logic.** When releases
  exceed the limit, the manifest references the archive page;
  the archive page itself is a separate template variant.
- **Does not maintain newsroom analytics.** Page-view and
  download analytics happen at deploy layer (Plausible, GA,
  CDN analytics).
- **Does not handle internationalized newsrooms.** Multi-
  language newsroom content would need per-locale boilerplate
  + per-locale release translations; out of scope.
