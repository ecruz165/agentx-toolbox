Generate `design/foundations/a11y.pen` and write the accessibility
tokens to `@theme`. Without a foundation, focus rings drift across
components, sr-only patterns get re-invented, and contrast checks
happen ad-hoc. This pulls the scattered concerns into one reference.

## Pre-flight

1. Read `product/strategy/_context.md` and `product/design/_context.md`.
2. Locate the `@theme` source CSS file.
3. Read `product/.pencil-brand.json` for the accent color (focus rings
   default to the accent, but can be overridden if accent has poor
   contrast against typical surfaces).

## Token set written to `@theme`

```css
@theme {
  /* Focus rings — visible keyboard-navigation indicator. Always
     2-color: an outer ring on a contrasting offset for visibility on
     any background. */
  --focus-ring-width:        2px;
  --focus-ring-offset:       2px;
  --focus-ring-color:        var(--color-accent-500);
  --focus-ring-color-offset: var(--color-background);
  --focus-ring-style:        solid;

  /* Composite for the standard visible-focus pattern */
  --focus-ring:
    0 0 0 var(--focus-ring-offset) var(--focus-ring-color-offset),
    0 0 0 calc(var(--focus-ring-offset) + var(--focus-ring-width)) var(--focus-ring-color);

  /* Screen-reader-only utility — visually hides content while
     keeping it in the accessibility tree */
  /* (Tailwind v4 generates the `sr-only` utility from this) */
  --a11y-sr-only-position:   absolute;
  --a11y-sr-only-clip:       rect(0, 0, 0, 0);
  --a11y-sr-only-overflow:   hidden;

  /* Touch-target minimum (WCAG 2.5.5 Level AAA = 44x44px) */
  --a11y-touch-target-min:   44px;

  /* Skip-link styling */
  --a11y-skip-link-bg:       var(--color-accent-700);
  --a11y-skip-link-fg:       var(--color-accent-50);
}
```

These tokens are the **foundation contract** — every component that
needs focus, screen-reader handling, or touch-target compliance reads
from them.

## Embedded prompt

> Build a Pencil page named **`Foundations / A11y`** for **{{brand}}**.
> Single 1440-wide canvas, 64px outer padding, 80px between sections.
> Render once on Light and once on Dark — focus rings must be visible
> on both.
>
> ### Section 1 — Focus rings
>
> A 4×2 grid showing focus-ring application across atom types:
> - Text input (focused) — ring around input perimeter
> - Button (default, focused) — ring around button
> - Link (focused) — ring around text run
> - Checkbox (focused, checked + unchecked) — ring around control
> - Radio (focused) — ring around control
> - Tab (focused) — ring around tab + selected indicator
> - Card (focused — clickable card pattern)
> - Image (focused — for image links / lightbox triggers)
>
> Each demonstrates the canonical `--focus-ring` composite shadow.
> Annotate each with its compound API call and a "tab order" note
> showing where in keyboard navigation it appears.
>
> ### Section 2 — Screen-reader-only content
>
> Demonstrate the `sr-only` utility: a visible button labeled "Edit"
> + the actual full label "Edit user profile for Jane Doe" hidden
> via sr-only. Side-by-side: the visible UI and the screen-reader
> rendition (rendered as plain text in a phone-shaped frame, with
> the SR-only content shown).
>
> Use cases section (a 3-column grid):
> - Icon-only buttons that need full labels
> - Form fields with implicit labels
> - Pagination context ("Page 3 of 12, currently on page 3")
>
> ### Section 3 — Touch-target minimums
>
> A measurement diagram showing the 44×44px minimum WCAG AAA
> touch target. Demonstrate compliant and non-compliant icon-button
> sizes:
> - ✅ 44×44 with 20px icon centered
> - ⚠️ 32×32 with 16px icon (tap-fail risk on mobile)
> - ✅ 32×32 button with 12px hit-area expansion via padded tap zone
>
> ### Section 4 — Skip links
>
> A demo of the canonical "Skip to main content" link — visually
> hidden by default, becomes visible on focus, jumps to `<main>`
> when activated.
>
> ### Section 5 — Color contrast guarantees
>
> A reference grid showing every accent-on-surface combination that
> components actually use, with computed AA / AAA badges. This is
> the single contrast reference for the system — components use these
> pairings, audit checks against this set.
>
> Pairings to render:
> - `accent-500` on `surface` (button bg + button text)
> - `accent-700` on `surface` (link color)
> - `content-1` on `accent-500` (text on solid button)
> - `content-1` on `surface` (default body text)
> - `content-2` on `surface` (secondary text)
> - `danger-500` on `surface` (error text)
> - `success-500` on `surface` (success text)
> - `warning-500` on `surface` (warning text)
>
> Each with WCAG AA (≥4.5:1 body, ≥3:1 large) and AAA badges. Failing
> pairings are flagged with a red border + "fails AA — fix the
> palette" callout.
>
> ### Section 6 — Reduced-motion behavior
>
> Cross-reference to `Foundations / Motion` — the reduced-motion
> contract is established there. Show one card here noting the

>

> - Demo atoms: `demo-{{role}}` (e.g. `demo-input-focused`)

## Component contract

After this foundation is written:
- Every interactive component must apply `--focus-ring` (or the
  composite shadow) on focus-visible state
- Every icon-only button must include sr-only text describing the
  action
- Every interactive element must hit `--a11y-touch-target-min` (44×44)
- Layouts must include a skip-link in the nav region
- The Phase 3 lint sweep checks for: missing focus-visible styling,
  icon-only buttons without sr-only labels, touch targets below 44px

## Audit hook

`/audit` Plane 3 (token drift) extends to verify the contrast
grid stays valid — if a palette change causes any pairing to fall
below AA, audit surfaces it as a warning.
