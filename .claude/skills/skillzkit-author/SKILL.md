# skillzkit-author

A scaffolder for skillzkit's catalog. Given the user's intent to add
a new command or workflow to the suite, this skill walks through the
authoring conventions, writes the artifact into a feature branch of
the kit clone, regenerates the catalog, and offers to PR it upstream
via `gh`.

## In scope

- **Commands** (single-shot tasks) under any persona namespace —
  `product/`, `engineer/`, `market/`, or extending an existing
  `core/*` namespace
- **Workflows** (multi-phase, persisted state) under
  `<persona>/workflows/` or `<persona>/<topic>/workflows/`

## Out of scope

- **Skills** — skills are maintainer-authored. Their frontmatter has
  to balance trigger-phrase precision across the whole router
  ecosystem and is too easy to get wrong without coordinating with
  existing routers. Direct the user to open an issue on the kit repo
  if they have a skill idea.
- **Tools / Integrations registry entries** — already covered by
  `/core:tools:declare` and `/core:integrations:declare`. Don't
  duplicate.
- **Frameworks bindings** (HeroUI, Storybook, etc.) — these are deep
  cross-cutting integrations; route to maintainer.
- **Authoring inside the user's project's `.claude/`** — that's a
  per-project hack, not a contribution. This skill targets the kit
  clone for upstream contribution.

## Pre-conditions

This skill assumes:

1. The user has a clone of `agentx-skillzkit` on disk. Detect it:
   - If `pwd` contains `package.json` with `name: @ecruz165/skillzkit`,
     you're already inside the clone.
   - Otherwise, ask the user for the path to their clone.
2. `gh` (GitHub CLI) is installed and authenticated (only required at
   the submission step — authoring works without it).
3. The clone's working tree is clean (no uncommitted changes that
   would tangle with the new branch). If dirty, surface and offer to
   stash before proceeding.

## Skillzkit architecture — what the agent needs to know

The kit's structure that authoring respects:

```
.claude/commands/
├── _context.md                    top-level orientation
├── core/                          cross-persona infra
│   ├── audit/                     audit dispatcher
│   ├── tools/                     local tool registry (npm, biome, etc.)
│   ├── integrations/              remote service registry (jira, github)
│   ├── workflows/                 workflow state machine
│   └── frameworks/                framework bindings (heroui, storybook)
├── product/                       product persona
│   ├── strategy/
│   ├── design/
│   ├── ux/
│   └── workflows/                 persona-level workflows
├── engineer/                      engineer persona
│   ├── architecture/
│   └── maintenance/
└── market/                        marketing persona
    ├── tone/  email/  ads/  social/  pr/  workflows/
.claude/skills/<name>/SKILL.md     agent-facing routers
catalog.json                       generated index of everything above
```

**Slug = path.** A file at `product/strategy/scaffold.md` becomes the
slug `product:strategy:scaffold`, invokable as `/product:strategy:scaffold`.

**Kinds** are inferred from path + frontmatter:

- **command** — anything not under `workflows/` and without `_` prefix
- **workflow** — file under any `workflows/` directory OR with
  `type: workflow` in frontmatter
- **context** — any file or path segment starting with `_`
  (e.g. `_context.md`, `_planes/product.md`); NOT runnable, just docs

**References** are auto-extracted from each body. Any string matching
`/foo:bar:baz` (slash + colon-separated identifier) is detected as a
reference to that slug. The catalog generator inverts this into
`referencedBy[]` so the dependency-graph features (suggest, install
cascade) work without per-file annotation.

**The catalog** is regenerated by `npm run catalog`. It walks
`.claude/commands/` and `.claude/skills/`, parses frontmatter, extracts
references, computes reverse edges, writes `catalog.json`. Every new
artifact must regenerate cleanly — if `npm run catalog` errors or
warns, fix before committing.

## Authoring flow

### Step 1 — Determine artifact kind and slug

Ask:

- **Kind**: command (single-shot task) or workflow (multi-phase,
  persisted state)?
- **Outcome**: one-line action phrase like "Define a persona" or
  "Apply a brand refresh" — this becomes the `outcome:` frontmatter
  and is what the TUI labels the row with.
- **Namespace**: which persona / topic does it belong under?
  - For commands: `<persona>:<topic>:[<sub-namespace>:]<slug>`
    e.g. `product:design:patterns:my-new-pattern`,
    `engineer:architecture:my-thing`,
    `market:email:my-template`
  - For workflows: `<persona>:[<topic>:]workflows:<slug>`
    e.g. `product:design:workflows:my-flow`,
    `engineer:maintenance:workflows:my-cycle`,
    `market:workflows:my-campaign` (market is flat)

Confirm the slug doesn't collide with an existing one — grep
`catalog.json` or `find .claude/commands -name "<leaf>.md"` to verify.

### Step 2 — Branch off main

```bash
cd <kit-clone-path>
git checkout main
git pull origin main           # only if a remote tracking branch exists
git checkout -b feat/<persona>-<slug>
```

Use `feat/` prefix for new artifacts. The branch name should hint at
what's being added — `feat/product-strategy-my-new-thing`,
`feat/engineer-maintenance-workflows-perf-audit`.

### Step 3 — Scaffold the file

Write the markdown to its target path. **Frontmatter conventions**:

```yaml
---
description: One-paragraph description that becomes the entry's catalog blurb. Should read as a complete sentence in the third person ("Generates the X..."). The TUI's right-pane and `skillzkit list` use this verbatim.
argument-hint: <subcommand> [args]    # optional; only for commands taking args
outcome: Define a persona              # imperative verb + outcome event/noun
allowed-tools: Read, Write, Edit, Bash # restrict if the command shouldn't have full tool access
---
```

For workflows, also include:

```yaml
estimatedDuration: 30-60 minutes       # human-readable
phases: 5                              # how many phases
prerequisites:                         # other workflows that must run first
  - product:strategy:workflows:greenfield
type: workflow                         # explicit kind tag (path-based detection also works)
```

**Body structure** — match the prevailing house style:

- Open with a one-paragraph orientation
- `## In scope` and `## Out of scope` sections so the agent knows
  when to fire / where to defer
- For commands: list subcommands as a markdown table when applicable
- For workflows: a phased outline with each phase's goal + which
  slash commands the phase invokes
- Reference other commands by their slash-form (`/foo:bar:baz`) — the
  catalog generator scrapes these into the `references[]` graph

### Step 4 — Validate

Run the catalog generator and check for warnings or errors:

```bash
npm run catalog
```

Expected: clean output with the new entry showing in the count.
**Lenient parse warnings** mean the YAML frontmatter has issues
(usually quoting). Fix them — they signal the next person will hit
parsing surprises.

Then check the new entry's references resolved:

```bash
node --input-type=module -e "
import { readFileSync } from 'node:fs';
const c = JSON.parse(readFileSync('catalog.json', 'utf8'));
const x = c.commands.find(c => c.slug === '<your-new-slug>');
console.log('references:', x?.references);
console.log('referencedBy:', x?.referencedBy);
"
```

If a referenced slug doesn't exist in the catalog, the regex still
captured it (it gets filtered to known-slugs only) — but the missed
references are silent. Eyeball the body for typos.

### Step 5 — Commit

```bash
git add .claude/commands/<path>     # or .claude/skills/<path>
git add catalog.json                # if it's tracked; check .gitignore
git commit -m "feat(<persona>): add <slug>

<one-line summary of what it does and why>

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>"
```

Don't `git add -A` — pick exactly the files you scaffolded. Stray
edits in the kit clone shouldn't ride along.

### Step 6 — Submit upstream (opt-in)

**Always confirm before pushing.** Authoring + committing locally is
the default; submission upstream is a separate gate. Ask the user:

> "Ready to push the branch and open a PR upstream? (You can also
> keep iterating locally and submit later.)"

If yes:

```bash
git push -u origin feat/<persona>-<slug>
gh pr create \
  --title "feat(<persona>): add <slug>" \
  --body "$(cat <<'EOF'
## Summary
<1-3 bullets describing what this adds>

## Why
<motivation — what problem does this solve, who benefits>

## Scope
- Files added: <list>
- Frontmatter fields used: <list>
- References to existing items: <list>

## Test plan
- [ ] `npm run catalog` runs clean
- [ ] New entry appears in `skillzkit list`
- [ ] If applicable: invocation produces expected output

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

Return the PR URL when `gh` finishes so the user can review.

## Anti-patterns

Do not:

- **Skip catalog regeneration.** A commit without a clean
  `npm run catalog` will break for the next person who pulls.
- **Author into the user's project's `.claude/`.** That's a
  per-project hack, not a contribution. New capabilities belong in
  the kit clone.
- **Add unconfirmed slugs to references.** If you write `/foo:bar:baz`
  but no such file exists, the catalog silently filters it out — so
  the new artifact looks fine but its dependency-graph entries are
  missing. Verify references resolve.
- **Push without explicit confirmation.** Authoring is local; pushing
  is opt-in.
- **Commit `node_modules/`, `dist/`, or local config.** `git add` only
  the scaffolded files (and `catalog.json` if tracked).
- **Use `git commit -a` or `git add -A`.** Stray working-tree edits
  should not ride along on a contribution PR.
- **Author skills here.** Skills go through maintainer review (they
  affect router precision across the whole kit). Open an issue
  instead.
- **Edit the TUI to surface the new artifact.** The TUI auto-discovers
  via the catalog; if the new entry has correct frontmatter and
  namespace, it surfaces without TUI changes.

## Example flow

> User: "I want to add a new command under product:strategy that
> generates a brand voice guide from a scraped competitor list."

Suggested response:

1. Confirm kind: **command** (single-shot, generates output)
2. Confirm slug: `product:strategy:competitor-voice-guide`
3. Confirm namespace fits: yes — strategy already has voice-related
   commands (`re-recommend`, `editorial`)
4. Branch: `feat/product-strategy-competitor-voice-guide`
5. Scaffold `.claude/commands/product/strategy/competitor-voice-guide.md`
   with frontmatter (description, outcome, allowed-tools, argument-hint)
   and a body that references `/product:strategy:editorial` and
   `/market:tone:explore`.
6. Run `npm run catalog` — expect 184 commands (was 183).
7. Commit on the feature branch.
8. Ask: PR now or keep iterating?
