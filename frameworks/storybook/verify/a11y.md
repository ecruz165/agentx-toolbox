---
description: Run accessibility audit on Storybook stories via axe-core. Surfaces WCAG violations, color contrast issues, missing labels, ARIA mistakes, keyboard accessibility gaps. Requires @storybook/addon-a11y to be in the manifest's addons.
argument-hint: [scope] [--severity critical|serious|moderate|minor] [--json] [--wcag-level a|aa|aaa]
allowed-tools: Read, Write, Edit, Bash
---

Run accessibility audit on Storybook stories using axe-core
via `@storybook/addon-a11y`. Surfaces WCAG violations grouped
by severity with actionable fix guidance.

Catches:
- Missing or invalid ARIA attributes
- Color contrast violations
- Missing form labels and associations
- Keyboard accessibility gaps (missing focus indicators,
  inaccessible interactive elements)
- Heading hierarchy violations
- Image alt text issues
- Landmark and region issues
- Document structure issues

## Phase 0: pre-flight

Per `frameworks/storybook/verify/_context.md`:
1. Storybook framework binding active
2. `.pencil-storybook.json` exists
3. Storybook running
4. Playwright tool available
5. `@storybook/addon-a11y` in manifest's addons:
   ```bash
   A11Y_ADDON=$(jq -r '.addons.a11y // empty' product/.pencil-storybook.json)
   if [ -z "$A11Y_ADDON" ]; then
     echo "@storybook/addon-a11y is required for a11y verification."
     echo "Install: npm install --save-dev @storybook/addon-a11y"
     echo "Then run /frameworks:storybook:init --update-addons to detect."
     exit 1
   fi
   ```

## Phase 1: scope resolution

Same pattern as `verify:health`. Resolves to story IDs.

## Phase 2: per-story a11y check

For each story, load the story in Playwright with axe-core
injected, run the audit, capture results:

```bash
LOCAL_URL=$(jq -r '.storybook.localUrl' product/.pencil-storybook.json)
WCAG_LEVEL="${WCAG_LEVEL:-aa}"  # default WCAG AA
TIMEOUT=$(jq -r '.screenshots.firstStoryTimeoutMs' product/.pencil-storybook.json)

cat > /tmp/run-axe.js <<'EOF'
const { chromium } = require('@playwright/test');
const fs = require('fs');

const STORY_URL = process.env.STORY_URL;
const TIMEOUT = parseInt(process.env.TIMEOUT || '30000');
const WCAG_LEVEL = process.env.WCAG_LEVEL || 'aa';

(async () => {
  const browser = await chromium.launch();
  const page = await browser.newPage();
  
  await page.goto(STORY_URL, { waitUntil: 'networkidle', timeout: TIMEOUT });
  await page.waitForTimeout(1000); // settle
  
  // Inject axe-core
  await page.addScriptTag({
    url: 'https://cdnjs.cloudflare.com/ajax/libs/axe-core/4.8.2/axe.min.js'
  });
  
  // Run audit
  const results = await page.evaluate(async (level) => {
    const tags = level === 'aaa' 
      ? ['wcag2a', 'wcag2aa', 'wcag2aaa', 'wcag21a', 'wcag21aa', 'wcag21aaa']
      : level === 'aa'
      ? ['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa']
      : ['wcag2a', 'wcag21a'];
    
    return await window.axe.run({
      runOnly: { type: 'tag', values: tags }
    });
  }, WCAG_LEVEL);
  
  console.log(JSON.stringify(results, null, 2));
  await browser.close();
})();
EOF

for STORY_ID in $STORY_IDS; do
  STORY_URL="${LOCAL_URL}/iframe.html?id=${STORY_ID}&viewMode=story"
  
  STORY_URL="$STORY_URL" TIMEOUT="$TIMEOUT" WCAG_LEVEL="$WCAG_LEVEL" \
    node /tmp/run-axe.js > "/tmp/a11y-${STORY_ID}.json" 2>/tmp/a11y-error.log
  
  EXIT=$?
  if [ "$EXIT" -ne 0 ]; then
    echo "✗ $STORY_ID — audit failed: $(cat /tmp/a11y-error.log)"
    FAILED+=("$STORY_ID")
    continue
  fi
  
  # Parse results
  VIOLATIONS=$(jq '.violations | length' "/tmp/a11y-${STORY_ID}.json")
  CRITICAL=$(jq '[.violations[] | select(.impact == "critical")] | length' "/tmp/a11y-${STORY_ID}.json")
  SERIOUS=$(jq '[.violations[] | select(.impact == "serious")] | length' "/tmp/a11y-${STORY_ID}.json")
  MODERATE=$(jq '[.violations[] | select(.impact == "moderate")] | length' "/tmp/a11y-${STORY_ID}.json")
  MINOR=$(jq '[.violations[] | select(.impact == "minor")] | length' "/tmp/a11y-${STORY_ID}.json")
  
  RESULTS+="$STORY_ID|$VIOLATIONS|$CRITICAL|$SERIOUS|$MODERATE|$MINOR"$'\n'
done
```

## Phase 3: severity filtering

When `--severity` provided, filter violations:

| Flag | Includes |
|------|----------|
| `--severity critical` | Critical only |
| `--severity serious` | Critical + Serious |
| `--severity moderate` | Critical + Serious + Moderate |
| `--severity minor` (default) | All |

## Phase 4: aggregate findings

Build a structured report grouping by:
1. Severity (critical → serious → moderate → minor)
2. Within severity, by violation rule (e.g., color-contrast,
   label, button-name)
3. Within rule, list affected stories with specific elements

```bash
# Example of grouping critical violations
jq -s '
  [.[] | .violations[] | select(.impact == "critical")]
  | group_by(.id)
  | map({
      rule: .[0].id,
      help: .[0].help,
      helpUrl: .[0].helpUrl,
      count: length,
      stories: [.[] | { storyId: .storyId, nodes: .nodes }]
    })
' /tmp/a11y-*.json > /tmp/a11y-critical.json
```

### JSON (`--json`)

Structured output with full violation details (rule ID,
help URL, affected nodes with HTML, suggested fix).

## Exit codes

- `0` — no violations at the configured severity
- `1` — moderate or minor violations only
- `2` — serious or critical violations

CI can treat exit code 0/1 as pass, 2 as fail.

## Cross-namespace effects

- **`product/design/foundations/a11y`** — when accessibility
  conventions are codified there, this command's report links
  back to the foundation for context
- **`engineer/maintenance/remediation/storybook-drift`** —
  drift routine includes a11y check across all stories on a
  cadence
- **MDX docs** — when `verify:fix` finds component-level a11y
  issues, surfaces "consider documenting in
  /frameworks:storybook:stories:doc accessibility section"

## What this command does NOT do

- **Modify component code.** Most a11y fixes require component
  changes (semantic HTML, ARIA attributes, contrast tweaks).
  This command surfaces violations; component changes route
  through manual edit or framework-binding rebuild.
- **Auto-fix story files for component-level violations.**
  Color contrast, label associations, etc. are component
  concerns. Story-level fixes (e.g., providing `aria-label`
  prop in story args) handled by `verify:fix` if applicable;
  this command only reports.
- **Validate axe-core's findings.** axe-core's rules are
  authoritative for what they check; this command surfaces
  what axe-core says.
- **Test interactive accessibility.** That's `verify:interactions`
  (keyboard navigation, focus management).

## Examples

```bash
# Audit all stories, all severities, WCAG AA
/frameworks:storybook:verify:a11y

# Just atoms
/frameworks:storybook:verify:a11y atoms

# Critical only
/frameworks:storybook:verify:a11y --severity critical

# Stricter WCAG level
/frameworks:storybook:verify:a11y --wcag-level aaa

# Specific component
/frameworks:storybook:verify:a11y Button

# CI integration
/frameworks:storybook:verify:a11y --json --severity serious > a11y.json
```
