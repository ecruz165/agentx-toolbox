# Storybook Stories — Sub-Namespace Context (`frameworks/storybook/stories/`)

> Read this in addition to `frameworks/storybook/_context.md` whenever any
> `/frameworks:storybook:stories:*` command runs.
>
> This sub-namespace owns deliberate story authoring — creating
> story files for components, finding components without
> stories, and generating MDX documentation. Drift work
> (missing stories detected during cycles, broken stories
> after upgrades) belongs to
> `engineer/maintenance/remediation/storybook-drift`.

## What this sub-namespace is for

Three commands:

- **`/frameworks:storybook:stories:gen`** — generate a story file for a
  specific component
- **`/frameworks:storybook:stories:gen-missing`** — find components
  without stories and generate them
- **`/frameworks:storybook:stories:doc`** — generate MDX documentation
  for a component

All three follow CSF (Component Story Format) conventions and
respect the project's manifest-defined organization, addons,
and provider stack.

## CSF format conventions

Stories use Component Story Format 3.x:

```typescript
import type { Meta, StoryObj } from "@storybook/react";
import { ComponentName } from "./ComponentName";

const meta = {
  title: "<derived from manifest>",
  component: ComponentName,
  parameters: {
    layout: "centered", // or "padded" / "fullscreen"
  },
  tags: ["autodocs"],
  argTypes: {
    // controls per prop
  },
} satisfies Meta<typeof ComponentName>;

export default meta;
type Story = StoryObj<typeof meta>;

export const Default: Story = {
  args: { /* minimal props */ },
};

// Variant stories follow
```

Hard rules across the sub-namespace:

- **`satisfies Meta<typeof Component>`** — use `satisfies`,
  not type annotation. Preserves narrow types for stories.
- **`type Story = StoryObj<typeof meta>`** — let StoryObj
  derive from the satisfied meta.
- **`tags: ["autodocs"]`** — explicit even when global default
  exists. Future readers shouldn't have to check `preview.js`.
- **Named exports for stories** — never default-export individual
  stories. The `default` export is the meta.
- **Realistic content** — pull from `.pencil-tone.json` and
  `.pencil-editorial.json` when present; never "Lorem ipsum"
  unless the component literally documents lorem patterns.

## Title derivation

Story titles are derived from
`componentOrganization.titlePattern` in the manifest. Examples
per convention:

### Atomic-design with sections

Pattern: `<Section>/<Category>/<Component>`

```
components/public/atoms/button/Button.tsx
  → title: "Public/Atoms/Button"

components/admin/molecules/data-table/DataTable.tsx
  → title: "Admin/Molecules/DataTable"

components/core/templates/layout/Layout.tsx
  → title: "Core/Templates/Layout"
```

### Atomic-design without sections

Pattern: `Core/<Category>/<Component>`

```
components/atoms/button/Button.tsx
  → title: "Core/Atoms/Button"
```

### Feature-based

Pattern: `<Feature>/<Component>`

```
components/auth/LoginForm/LoginForm.tsx
  → title: "Auth/LoginForm"
```

### Flat

Pattern: `Components/<Component>`

```
components/Button/Button.tsx
  → title: "Components/Button"
```

### Custom

Per the manifest's user-defined mapping. The pattern explicitly
specifies how directories translate to title segments.

## Variant generation

Story files generated by this sub-namespace include variants
based on detected component shape. The detection looks at the
component's prop interface and renders applicable variants:

### Size variants

Triggered by props like `size`, `scale`, `dimension` with enum
types:

```typescript
export const AllSizes: Story = {
  render: (args) => (
    <div className="flex gap-4 items-center">
      <ComponentName {...args} size="sm" />
      <ComponentName {...args} size="md" />
      <ComponentName {...args} size="lg" />
    </div>
  ),
};
```

### Color variants

Triggered by props like `color`, `variant`, `intent` with
color-shaped enum values:

```typescript
export const AllColors: Story = {
  render: (args) => (
    <div className="flex gap-4">
      <ComponentName {...args} color="default" />
      <ComponentName {...args} color="primary" />
      <ComponentName {...args} color="secondary" />
      <ComponentName {...args} color="success" />
      <ComponentName {...args} color="warning" />
      <ComponentName {...args} color="danger" />
    </div>
  ),
};
```

### Variant variants (style modes)

Triggered by `variant` prop with style-shaped values
(`solid` / `bordered` / `flat` / `light` / `ghost` / etc.):

```typescript
export const AllVariants: Story = {
  render: (args) => (
    <div className="flex gap-4">
      <ComponentName {...args} variant="solid" />
      <ComponentName {...args} variant="bordered" />
      <ComponentName {...args} variant="flat" />
    </div>
  ),
};
```

### State variants

Triggered by boolean props like `disabled`, `loading`, `error`,
`empty`, `selected`:

```typescript
export const Disabled: Story = {
  args: { disabled: true },
};

export const Loading: Story = {
  args: { loading: true },
};
```

### Compound component variants

For compound components (e.g., HeroUI v3 patterns), generate
stories showing the compound usage:

```typescript
export const Compound: Story = {
  render: () => (
    <Component>
      <Component.Trigger>Open</Component.Trigger>
      <Component.Content>Content here</Component.Content>
    </Component>
  ),
};
```

Detection: the component file exports nested members
(`Component.Trigger`, `Component.Content`).

### Dark mode

When the manifest's provider stack includes a theme provider
(e.g., `ThemeProvider`, `next-themes`), the autodocs page picks
up the theme toolbar. Don't generate explicit dark mode stories
unless the component has dark-mode-specific differences worth
documenting.

## MSW handler wiring

Components that fetch data need MSW handlers in their stories.
Detection: the component imports from a fetching library
(`fetch` / `axios` / `@tanstack/react-query` / `useSWR`).

When MSW is in the manifest's addons:

```typescript
import { myHandlers } from "./mocks/handlers";

const meta = {
  title: "...",
  component: ComponentName,
  parameters: {
    msw: {
      handlers: myHandlers,
    },
  },
} satisfies Meta<typeof ComponentName>;
```

The handler file is conventionally at
`<component-dir>/mocks/handlers.ts`. If it doesn't exist, the
gen command surfaces it as a follow-up:

> Component fetches data; needs MSW handlers. Suggested
> location: <component-dir>/mocks/handlers.ts. Generate
> handler stub now? [Y/n]

## Interaction tests

When `@storybook/test` is in the manifest's addons, interactive
components get play-function stories:

```typescript
import { expect, userEvent, within } from "@storybook/test";

export const ClickHandler: Story = {
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);
    const button = canvas.getByRole("button");
    await userEvent.click(button);
    await expect(button).toHaveAttribute("aria-pressed", "true");
  },
};
```

Detection: the component has interactive props (`onClick`,
`onChange`, `onSubmit`, `onSelect`).

## Provider config in story args

When the manifest's provider stack includes a theme or i18n
provider, stories don't re-wrap (the provider is already in
`preview.js` decorators). But when story args need to interact
with provider state (e.g., simulating a logged-in user, a
specific theme), stories use the provider's API:

```typescript
import { useTheme } from "next-themes";

export const ForcedDark: Story = {
  decorators: [
    (Story) => {
      const { setTheme } = useTheme();
      useEffect(() => setTheme("dark"), [setTheme]);
      return <Story />;
    },
  ],
};
```

Generally avoid this pattern — the theme toolbar handles
light/dark switching globally. Use only when stories need
specific provider state that the toolbar doesn't expose.

## Component analysis

Before generating, the sub-namespace's commands analyze the
component file to extract:

- **Props interface** — read TypeScript interface or
  JSDoc-typed props; identify required vs optional, defaults,
  enum values
- **Variants** — props matching size/color/variant patterns
- **Dependencies** — imports from data-fetching libraries,
  state managers, providers
- **Atomic level** — derived from filesystem location per
  manifest's `componentOrganization.hierarchy`
- **Existing stories** — check for sibling `.stories.tsx` /
  `.stories.ts` files (gen prompts to update vs replace; this
  is the gen-missing exception path that delegates back to gen
  with `--update`)
- **Compound exports** — sub-components exposed via
  `Component.Subcomponent` pattern

## Suite-wide content sources

Generated stories prefer content from suite-managed sources:

### Tone and editorial

When `product/.pencil-tone.json` exists, the gen command reads
voice attributes (formality, warmth, technical density) and
uses them when generating story labels, button text, and
sample copy.

When `product/.pencil-editorial.json` exists, the command reads
writing conventions (sentence length preferences, capitalization
rules, terminology) and applies them.

### Brand

When `product/.pencil-brand.json` exists, the command reads brand
name, product name, and key value propositions for use in
realistic story content.

### Domain

When component prop names hint at domain (e.g., `studentName`,
`scholarshipAmount`, `enrollmentDate`), the command derives
realistic values from the project's domain rather than
generic placeholders. Detection: prop name patterns +
manifest's componentOrganization.featureSections names.

## Lint after generation

Every story file generated or modified runs the manifest's
lint command after writing:

```bash
LINT_CMD=$(jq -r '.lint.command' product/.pencil-storybook.json)
LINT_DIR=$(jq -r '.lint.workingDir' product/.pencil-storybook.json)
(cd "$LINT_DIR" && $LINT_CMD)
```

Lint failures are surfaced before the command considers itself
complete. The user can fix and re-run, or accept and push fixes
in a follow-up commit.

## Anti-patterns

- **Hardcoded titles** — never write story titles literally;
  always derive from componentOrganization patterns
- **Lorem ipsum content** — pull from tone/editorial/domain
  sources or use realistic equivalents
- **Bypassing autodocs** — every story file should include
  `tags: ["autodocs"]` even when global default applies
- **Inline default-only stories** — generate at least 2-3
  meaningful variants when applicable; single-default stories
  often signal incomplete coverage
- **MSW handlers inlined in story file** — handlers belong in
  `<component-dir>/mocks/handlers.ts` and get imported. Inline
  handlers are anti-pattern for reuse and discoverability.
- **Re-wrapping providers from preview.js** — providers in
  `preview.js` apply globally; don't re-wrap unless stories
  need specific provider state.
- **Skipping interaction tests for interactive components** —
  when `@storybook/test` is available and the component has
  click/change/submit handlers, generate at least one play
  function.
